Da história ao código, com IA e governança.

Este repositório é uma fábrica de scaffolding corporativo: padrões, agentes, skills e automações que geram e evoluem projetos .NET reais — partindo das histórias no seu tracker até o Pull Request, sempre dentro da Clean Architecture.

.NET 10 Clean Architecture UseCase Dispatcher (sem MediatR pago) Banco plugável: Oracle · SQL Server · PostgreSQL · MySQL Serilog + OpenTelemetry Polly Hangfire Kafka · SQS · RabbitMQ · MQTT Azure DevOps · GitHub · GitLab

3 Banco de dados (plugável)

Antes fixo em Oracle, agora selecionável por configuração — mesma filosofia das filas. O motor é detalhe de Infrastructure, atrás das ports da Application; trocar de banco não muda Domain/Application nem o fluxo. Padrão recomendado: Oracle. (ADR-0013, substitui o ADR-0004.)

🔶
OracleOracle.EntityFrameworkCore · driver ManagedDataAccess.Core
🟦
SQL ServerMicrosoft.EntityFrameworkCore.SqlServer
🐘
PostgreSQLNpgsql.EntityFrameworkCore.PostgreSQL
🐬
MySQLPomelo.EntityFrameworkCore.MySql

Como escolher

No scaffolding do projeto (parâmetro do /create-project, padrão oracle):

crie um projeto com nome Billing com banco PostgreSQL e fila RabbitMQ

E em runtime, por configuração no projeto gerado:

// appsettings.json
{ "Database": { "Provider": "PostgreSql" } }   // Oracle | SqlServer | PostgreSql | MySql

O que muda (e o que não muda)

Não muda

Domain, Application, as ports (ex.: IPedidoRepository), os casos de uso e os testes. O código de negócio é o mesmo em qualquer banco.

Muda (só Infrastructure)

O provider EF Core referenciado, a connection string e os scripts de migração no dialeto do banco, em db/<provider>/migrations.

Scripts de migração por provider

db/
└── <provider>/                 # oracle | sqlserver | postgresql | mysql
    ├── migrations/
    │   ├── V0001__create_schema.sql   # up
    │   └── U0001__create_schema.sql   # down (rollback)
    └── seed/

Crie scripts com /create-db-script (gera no dialeto do provider) e valide com scripts/validate-db-scripts.ps1.

Resiliência: chamadas ao banco usam a pipeline Polly database (timeout + retry transitório + circuit breaker), igual para qualquer provider. Testes: integração via Testcontainers — há imagem para os quatro bancos.
Convenções: Oracle em UPPER_SNAKE_CASE; PostgreSQL/MySQL em snake_case; SQL Server em PascalCase/snake_case (consistente). Detalhes e equivalências de tipos em docs/standards/database.md.

4 Tipos de história

Nem todo trabalho é feature de negócio. O template reconhece dois tipos.

💼 Negócio

Valor para o usuário. Critérios de aceite comportamentais (Given/When/Then).

Skill: /create-feature · Fluxo: /approve-architecture → /create-usecase

🏗️ Técnica

Plataforma/arquitetura: setup do projeto, arquitetura base, pipeline, observabilidade. Critérios de aceite técnicos.

Skill: /create-tech-story · Fluxo: /create-project e/ou /approve-architecture (sem use cases de negócio)

Como a IA reconhece: uma história importada é técnica se tiver uma tag/label de storyKinds.technicalLabels (default: historia-tecnica, tecnica, tech, arquitetura, infra).

5 Integração com o tracker

As histórias vêm de um tracker plugável. Trocar de origem é mudança de configuração, não de fluxo.

ProviderMecanismoCredencial (variável de ambiente)
Azure DevOpsREST dev.azure.com (Work Items)AZDO_PAT (Work Items: Read & Write)
GitHubgh CLI ou REST api.github.comGITHUB_TOKEN (escopo repo)
GitLabREST /api/v4GITLAB_TOKEN (read_api / api)

Configuração — .claude/tracker.config.json

// Placeholders versionados; dados reais vão em tracker.config.local.json (fora do git).
// Segredos NUNCA aqui — só variáveis de ambiente.
{
  "provider": "azure",
  "azure": { "org": "minha-org", "project": "MeuProjeto" },
  "github": { "repo": "" },
  "gitlab": { "host": "https://gitlab.com", "projectId": "" },
  "taskSync": { "enabled": true, "azureTaskType": "Task" },
  "storyKinds": { "technicalLabels": ["historia-tecnica", "tecnica", "arquitetura"] }
}

Importar: "importe o work item 1234" → gera docs/features/AZ-1234-<slug>.md com problema, contexto e critérios de aceite, e atualiza o backlog.

9 Passo a passo prático

Do zero ao PR. O exemplo usa Azure DevOps como tracker.

0 · Pré-requisitos

.NET 10 SDK, Git, Docker (para o banco/filas nos testes), e a credencial do tracker em variável de ambiente (AZDO_PAT).

1 · Crie o projeto

Diga: "crie um projeto com nome Billing com banco PostgreSQL e fila RabbitMQ". O banco é plugável (Oracle padrão · SQL Server · PostgreSQL · MySQL). Gera a solução, builda vazia e passa nos testes de arquitetura.

2 · Traga a história

Diga: "importe o work item 1234". Gera docs/features/AZ-1234-<slug>.md com os critérios de aceite. (Ou "crie uma feature…" do zero.)

3 · Aprove a arquitetura

Diga: "abra arquitetura da feature AZ-1234". Produz o design, os ADRs e cria as tasks no Azure ligadas à história.

4 · Implemente

Diga: "implemente o use case ConfirmarCobranca". Domain → Application → Infrastructure → Api, com Polly, OTel e logs estruturados.

5 · Teste

Diga: "crie os testes da feature AZ-1234". Cada critério de aceite vira ≥1 teste.

6 · Revise e abra o PR

Diga: "revise o PR". Roda os gates e os revisores; o PR referencia #1234 e fecha o work item.

10 Modos de execução

Você escolhe quanto a IA encadeia sozinha entre as etapas.

Por etapa

Uma frase por vez; você revisa entre cada passo. Padrão recomendado.

Até um checkpoint

"crie a feature e abra a arquitetura, depois pare para eu aprovar".

Ponta a ponta

"crie a feature e leve até o PR sem parar".

Dica: pare nos dois pontos onde a decisão é sua — feature (o quê) e arquitetura (o como). Depois, código → testes → review pode correr em sequência.

11 Guia do Product Owner

Como escrever histórias no Azure DevOps para que sejam importáveis e implementáveis pela IA.

Campo Description

História
Como <persona>, quero <objetivo>, para <benefício>.

Regras de negócio
- <regra clara e verificável>

Requisitos não funcionais
- Performance / Segurança-PII / Confiabilidade / Observabilidade

Dependências / Integrações
- <sistema, fila, API>

Campo Acceptance Criteria (vira teste)

Cenário: Confirmação dentro do limite
  Dado um cliente com limite disponível de 1000
  E um pedido no valor de 800
  Quando o analista confirma o pedido
  Então o pedido fica "Confirmado"

Cenário: Bloqueio por crédito insuficiente
  Dado um cliente com limite de 500
  E um pedido de 800
  Quando o analista confirma
  Então a confirmação é recusada com "Crédito insuficiente"
Definition of Ready: Title claro · Description com regras de negócio e NFRs · Acceptance Criteria em Given/When/Then cobrindo sucesso + validação + erro · estimada e priorizada.

Documento completo no repositório: docs/standards/escrita-de-historias.md. Dica: salve esse formato como Work Item Template do time no Azure DevOps.

13 FAQ

Preciso ficar conduzindo agente por agente?

Não. Dentro de cada skill os agentes rodam em sequência sozinhos. Você só decide quanto encadear entre as etapas (ver Modos de execução).

Onde ficam as histórias?

Como documentos versionados em docs/features/, geradas do tracker (/import-story) ou do zero (/create-feature). É de lá que a IA lê para implementar.

Como troco o banco de dados de um projeto?

Escolha o provider no /create-project (ou Database:Provider). Domain/Application/testes não mudam — só o provider EF Core, a connection string e os scripts em db/<provider>.

E os segredos (PAT/tokens)?

Nunca no código nem no tracker.config.json — somente em variáveis de ambiente.

Posso trocar de tracker depois?

Sim. Mude provider no tracker.config.json; o fluxo e os agentes permanecem iguais.
